## 当前项目可复用与需重写

### 可复用

- Electron 桌面壳和打包流程。
- React 页面、侧边栏、设置页、日志面板、Toast。
- IPC 桥接模式。
- 全局快捷键启停能力。
- 设置持久化和运行统计。
- `nut-js` 鼠标键盘控制，但需要先实测九阴真经是否接受这种软件模拟输入。
- 截图、OCR、OpenCV 模板匹配的基础能力。
- 游戏浮窗思路。

### 需要新建或重写

- `LCUManager` 这类 LOL 客户端 API 连接层不适用于九阴。
- 旧固定坐标、阶段、单位、装备和搭配协议已从运行层移除；九阴需要重新建立自己的坐标和任务协议。
- 旧操作器已从运行层移除；阶段 0 已新增通用 `GameOperator`，并实现空状态的 `JiuYinOperator`。
- `StrategyService` 需替换为九阴任务策略系统。
- `GameStageMonitor` 需替换为九阴场景/任务状态监视器。
- 旧搭配系统已移除；后续需新建任务配置、路径配置、技能配置、资源模板配置。
- `public/resources` 需新增九阴模板资源和配置。
- 需新增输入后端抽象：默认实现 `NutInputBackend`，预留 `HardwareInputBackend`，避免后续发现软件输入不可用时大面积返工。

## 输入后端与外部依赖说明

### 默认输入后端：nut-js

- 当前项目已经依赖 `@nut-tree-fork/nut-js`，可继续作为第一版默认输入后端。
- 第一阶段必须做输入探针，确认九阴真经窗口内是否接受：
  - 前台鼠标点击。
  - 鼠标移动和拖拽。
  - 普通按键。
  - 组合键。
  - 团练/授业类提示按键。
- 如果 `nut-js` 在九阴窗口内可用，则继续按原路线开发团练、授业、生活和日常功能。
- 如果 `nut-js` 在九阴窗口内不可用，则暂停业务功能开发，优先完成 `InputBackend` 抽象并评估硬件输入后端。

### 可选硬件后端：幽灵键鼠

- “幽灵键鼠”通常是插 USB 的外部 HID 设备，系统会将它识别为真实键盘/鼠标。
- 脚本向硬件发送指令，再由硬件向系统发出键鼠事件，游戏侧更接近真实外设输入。
- 第一版不要求购买或接入幽灵键鼠；只有在软件输入不可用、且用户明确提供设备和协议资料时，再考虑实现 `HardwareInputBackend`。

### 参考但不默认接入：大漠插件

- 大漠插件常见于按键精灵/Windows 自动化生态，提供找图、找色、OCR、窗口绑定、鼠标键盘模拟等能力。
- 本项目第一版不接入大漠插件，原因：
  - Windows/COM/授权/注册流程复杂。
  - 与当前 Electron + TypeScript 技术栈不一致。
  - 容易引入窗口绑定、后台操作、驱动或反检测争议。
- 大漠只作为理解 PC助手实现形态和功能覆盖面的参考，不作为默认技术路线。

## 设计原则

- 只走“图色识别 + OCR + 鼠标键盘”的白盒可维护路线。
- 所有自动化都要求游戏窗口可见、非最小化、画面完整。
- 先验证输入是否真实生效，再开发业务功能；输入后端验证失败时，不继续堆功能。
- 不做反检测、虚拟机绕过、驱动绕过、封包、内存、注入、后台绑定窗口、关闭安全软件或加入白名单等规避类方案。
- 任务默认支持人工接管，F10/F12 等快捷键必须可靠停止，并避免占用游戏常用的 F1~F9。
- 每个功能都先做“单账号、单窗口、单流程”，稳定后再考虑多窗口调度。
- 对涉及银两、竞拍、交易、邮件的功能默认降级为提醒或半自动，避免误操作造成损失。

  - 支持任务失败策略：跳过、重试、停止并提醒。
- 验收：
  - 能串联 2 个已稳定功能。
  - 每个节点都有日志、耗时、失败截图。

### 阶段 10：高风险功能谨慎处理

- 世界竞拍、邮件批量处理、交易/银两相关功能只做提醒或半自动确认。
- 若实现，必须增加：
  - 金额上限。
  - 二次确认。
  - 干跑模式。
  - 操作前截图留档。
  - 用户可撤销或可中断的等待窗口。
